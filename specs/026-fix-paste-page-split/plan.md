# Implementation Plan: Fix Paste Content Page Splitting

**Branch**: `026-fix-paste-page-split` | **Date**: 2026-03-24 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/026-fix-paste-page-split/spec.md`

## Summary

When content is pasted into the canvas editor, the overflow detection logic fails to split content across pages — unlike typing, which correctly splits. The root cause is threefold: (1) overflow detection checks cursor position instead of content bottom, (2) only one block is extracted per overflow event, and (3) the hysteresis gate blocks further splits. The fix changes the overflow detector to measure actual content extent, extract ALL overflowing blocks at once, and allow natural cascade across pages.

## Implementation Phases

### Phase 1: Fix overflow detection to measure content bottom (FR-001, FR-005, FR-008)

**File**: `src/components/canvas/canvas-page.tsx` (lines 426–518)

**What changes**: Replace cursor-based overflow measurement with content-bottom measurement.

**Current code** (line 432):

```typescript
const coords = ed.view.coordsAtPos(ed.state.selection.from);
```

**New code**:

```typescript
// Measure the bottom of the LAST position in the document,
// not the cursor. This detects overflow after paste (cursor may
// be anywhere) and after focusPage sets cursor to start.
const lastPos = ed.state.doc.content.size;
const coords = ed.view.coordsAtPos(lastPos > 0 ? lastPos - 1 : 0);
```

**Why**: `selection.from` tracks the cursor, which after paste or `focus('start')` may be at the top of the page. Checking the end of the document detects overflow regardless of cursor position.

**Interview concept**: This is a classic "measure what matters" debugging pattern. The symptom was "paste doesn't split" but the root cause was an incorrect measurement point — the overflow detector was looking at the wrong position.

---

### Phase 2: Extract ALL overflowing blocks at once (FR-001, FR-002, FR-003)

**File**: `src/components/canvas/canvas-page.tsx` (lines 440–451)

**What changes**: Replace the single-last-block extraction with a bulk extraction from the first overflowing block onwards.

**Current code** (multi-block case):

```typescript
if (doc.childCount > 1) {
  const lastChild = doc.lastChild!;
  const lastNodeJson = lastChild.toJSON();
  const nodeFrom = doc.content.size - lastChild.nodeSize;
  const nodeTo = doc.content.size;
  ed.chain().deleteRange({ from: nodeFrom, to: nodeTo }).run();
  ed.commands.blur();
  onTextOverflowRef.current?.(pageIdRef.current, {
    type: 'doc',
    content: [lastNodeJson],
  });
}
```

**New code**:

```typescript
if (doc.childCount > 1) {
  // Find the first block whose bottom edge exceeds PAGE_HEIGHT
  let splitChildIndex = doc.childCount; // default: no overflow
  let posAfterPrev = 0;
  for (let i = 0; i < doc.childCount; i++) {
    const child = doc.child(i);
    const endPos = posAfterPrev + child.nodeSize;
    try {
      const childCoords = ed.view.coordsAtPos(endPos - 1);
      const childBottom = childCoords.bottom - layerRect.top;
      if (childBottom > PAGE_HEIGHT) {
        splitChildIndex = i;
        break;
      }
    } catch {
      break;
    }
    posAfterPrev = endPos;
  }

  // Ensure at least one block stays on the current page
  if (splitChildIndex === 0) splitChildIndex = 1;

  if (splitChildIndex < doc.childCount) {
    // Extract blocks from splitChildIndex onwards
    const overflowNodes: unknown[] = [];
    let deleteFrom = 0;
    for (let i = 0; i < doc.childCount; i++) {
      if (i === splitChildIndex) deleteFrom = posAfterPrev;
      if (i >= splitChildIndex) {
        overflowNodes.push(doc.child(i).toJSON());
      }
      posAfterPrev += doc.child(i).nodeSize; // only needed if loop continues
    }

    // Actually compute deleteFrom by walking up to splitChildIndex
    let from = 0;
    for (let i = 0; i < splitChildIndex; i++) {
      from += doc.child(i).nodeSize;
    }

    ed.chain().deleteRange({ from, to: doc.content.size }).run();
    ed.commands.blur();
    onTextOverflowRef.current?.(pageIdRef.current, {
      type: 'doc',
      content: overflowNodes,
    });
  }
}
```

**Why**: Extracting all overflowing blocks in one operation means the current page fits within bounds after a single split event. The overflow content goes to the next page, which may itself cascade if it also overflows.

**Important detail**: We ensure `splitChildIndex >= 1` so the current page always retains at least one block. This prevents an empty page from being left behind.

### Phase 3: Reset hysteresis gate and enable cascade (FR-002)

**File**: `src/components/canvas/canvas-page.tsx` (line 437, line 512)

**What changes**: After a successful split, reset the `overflowNotifiedRef` gate. This allows the next `onUpdate` (from the receiving page's cascade) to detect its own overflow.

The gate was originally needed to prevent infinite loops when the cursor sits at the boundary during typing. With the new approach (checking content bottom and extracting all overflow blocks), the loop risk is eliminated because:

- After split, the remaining content fits within PAGE_HEIGHT
- The `onUpdate` fires again (from the delete operation), but content bottom is now within bounds
- So overflow detection exits early

**Change**: After the split operation sends overflow content, reset the gate:

```typescript
overflowNotifiedRef.current = false;
```

Additionally, the `else if (cursorY < PAGE_HEIGHT - 100)` hysteresis reset (line 512) should now check content bottom instead of cursor position for consistency:

```typescript
} else if (contentBottom < PAGE_HEIGHT - 100) {
  overflowNotifiedRef.current = false;
}
```

**Why**: The cascade from page N → page N+1 → page N+2 relies on each page's overflow detection firing independently. Without resetting the gate, a page that has already split once would never split again, even if it received new overflow content.

### Phase 4: Handle single-block overflow during paste (FR-003)

**File**: `src/components/canvas/canvas-page.tsx` (lines 452–510)

**What changes**: The existing single-block split logic (word-boundary split) works correctly for the case where a single paragraph overflows. No changes needed to this logic itself.

However, after the split, the overflow (second half of the paragraph) goes to the next page. If that paragraph is very long (multiple pages), the next page also overflows → cascade. The cascade works because:

1. Next page receives content via `setContent` in `focusPage`
2. `onUpdate` fires on next page
3. Content bottom > PAGE_HEIGHT → single-block split logic runs again
4. Repeat until content fits

**Verify**: The single-block split does not need modification, but we must verify the cascade works by ensuring:

- `overflowNotifiedRef` is `false` on the receiving page (it is — each page has its own ref)
- `focusPage` calls `setContent` which triggers `onUpdate` (it does — `setContent` without `{ emitUpdate: false }`)

### Phase 5: Adjust parent overflow handler for bulk content (FR-002, FR-006)

**File**: `src/components/canvas/canvas-editor.tsx` (lines 844–889)

**What changes**: The `handleTextOverflow` callback and `focusPage` already handle arbitrary overflow content size. When the overflow content contains multiple blocks, `focusPage` merges them at the start of the next page's content. No structural changes needed.

**Minor adjustment**: After the cascade completes (all pages settled), the cursor should be on the final page where content was placed. Currently `focusPage` calls `editor.commands.focus('start')` on each intermediate page during cascade. This is acceptable — the final `focusPage` call will leave the cursor on the last target page.

**Scroll behavior**: `scrollToPage(pageId)` is called in `handleTextOverflow` for the first overflow. During cascade, subsequent overflow handling on downstream pages also calls `scrollToPage`. The final scroll position will be on the last page that received overflow content. This is reasonable UX — the user sees where their pasted content ends.

## Design Decisions

### Why cascade instead of pre-computation?

Pre-computing the page distribution of pasted content would require knowing rendered heights of each block before rendering. Block heights vary significantly (headings are taller, math blocks vary, list indentation changes height). The only reliable measurement is after rendering. The cascade approach renders → measures → splits → repeats, ensuring accuracy at the cost of sequential processing (~100ms per page).

### Why not a custom `handlePaste` interceptor?

TipTap's `handlePaste` fires before content is inserted. At that point, we can see the ProseMirror `Slice` from the clipboard, but we can't measure rendered heights. We'd have to estimate, which would be inaccurate. Instead, we let TipTap insert the content normally and then handle overflow in `onUpdate`, which is the same code path as typing. This keeps the logic unified.

### Why reset the hysteresis gate?

The original gate prevented infinite re-triggering when the cursor sits at the page boundary during typing. With the new approach (extracting ALL overflow blocks at once), the page is guaranteed to be within bounds after a split. The `onUpdate` from the delete operation sees content bottom < PAGE_HEIGHT and does not re-trigger. So the gate is no longer needed for preventing loops, but we keep it as a safety measure and reset it after successful splits.

### Undo limitation (FR-007)

TipTap's undo stack is per-editor (per-page). A paste-and-split spans multiple editors. Undoing on the first page restores its content but does not remove pages created during overflow. This is the **same behavior as typing-based page splits** — it's a pre-existing limitation, not a new one. True cross-page atomic undo would require a custom undo system wrapping all page editors, which is a separate feature.
